# เฉลยแบบเห็นภาพ — กล่องขั้นตอนสี + ไดอะแกรมประกอบ (คู่มือทำต่อบทอื่น/วิชาอื่น)

เริ่มทำครั้งแรกกับบทที่ 1 (ทักษะพื้นฐานปฏิบัติการเคมี) 17-19 ก.ค. 2569 หลังผู้ใช้ทดลองให้เด็กดู
เฉลยเดิมแล้วไม่ค่อยเข้าใจ — เอกสารนี้สรุปธรรมเนียม/โค้ด/เวิร์กโฟลว์ที่ใช้ เพื่อทำบทอื่นหรือวิชาอื่น
ต่อได้โดยไม่ต้องคิดใหม่ทั้งหมด

## หลักการสำคัญที่สุด: ไม่ต้องเปลี่ยน schema

เฉลย (`explanation` field) ยังเป็น **Markdown string เดียว** เหมือนเดิมทุกประการ — ไม่มีการเพิ่ม
field ใหม่ใน JSON, ไม่มีการแก้ตาราง Supabase, ไม่มีการเปลี่ยน view/RPC ทั้งกล่องสีและไดอะแกรม
ทำได้ด้วย **ธรรมเนียมการเขียน** + **การเรนเดอร์ที่ฉลาดขึ้น** เท่านั้น

## ธรรมเนียมป้าย (label) → กล่องสี

เขียนย่อหน้าเดี่ยวๆ ขึ้นต้นด้วย `**ป้าย:**` (ตัวหนา ตามด้วย `:` หรือไม่ก็ได้) แล้วเว้นบรรทัดว่าง
คั่นย่อหน้าก่อน-หลัง ระบบจะจับคู่ป้าย → กล่องสีให้อัตโนมัติทั้งเว็บและ PDF:

| ป้าย (regex จับคำที่มี) | กล่อง | สี |
|---|---|---|
| หลักคิด, วิธีคิด, หลักการ, นิยาม, สูตรลัด, วิธีจำ, เกร็ดเพิ่ม | `cal-info` | ฟ้า `#2563eb` |
| `ขั้นที่ N` (ตัวเลขอะไรก็ได้) | `cal-step` (มีวงกลมเลขขั้น) | เขียวมรกต `#0d9488` |
| ตรวจสอบ, ขึ้นต้นด้วย "ตรวจ", สรุป, สังเกต, ข้อสังเกต | `cal-check` | เขียว `#059669` |
| ระวัง, ตัวลวง | `cal-warn` | ส้ม `#ea580c` |
| **ตอบ** (พอดีคำ ไม่ต้องมีคำอื่นปน) | `cal-answer` | เขียวอ่อน (เด่นสุด) |

**กติกาใหม่ที่เพิ่มจากบทที่ 1 (บังคับทุกข้อจากนี้ไป):** ทุกเฉลยต้องปิดท้ายด้วย
`**ตอบ:** ข้อ ก/ข/ค/ง. <ข้อความคำตอบเต็มจากอาร์เรย์ choices>` เป็นบรรทัดสุดท้าย — ก่อนหน้านี้
เฉลยจบด้วยประโยคลอย "ดังนั้นคำตอบคือ..." ซึ่งเด็กมองไม่เห็นเป็นจุดจบชัดเจน กล่องเขียวปิดท้าย
ทำให้รู้ทันทีว่า "อ่านจบแล้ว นี่คือคำตอบ"

ตรวจสอบว่าตัวอักษร ก/ข/ค/ง ตรงกับ `correct_index` (0/1/2/3) และข้อความตรงกับ
`choices[correct_index]` เป๊ะ — ผู้ตรวจอิสระต้องเช็คจุดนี้ทุกครั้ง

โค้ดที่ implement ธรรมเนียมนี้ (ต้องแก้ทั้งคู่ให้ตรงกันถ้าจะเปลี่ยน regex):
- **PDF**: `scripts/build-worksheet.mjs` ฟังก์ชัน `styleCallouts()` (มีอยู่แล้วก่อนงานนี้)
- **เว็บ**: `src/components/MathMarkdown.tsx` ฟังก์ชัน `calloutInfo()` + `walk()`
  (เพิ่มใหม่ในงานนี้ — เป็น rehype-plugin เล็กๆ พอร์ต logic เดียวกับฝั่ง PDF โดยไม่พึ่ง
  dependency ใหม่ เก็บ 2 ชุดแยกไฟล์เพราะคนละ runtime — Node ESM script vs Next/TS component)
- CSS สี: `.cal-*` ใน `src/app/globals.css` (เว็บ) กับตัวแปร `--info/--check/--warn` +
  `.cal-*` ใน `<style>` ของ `build-worksheet.mjs` (PDF) — ใช้โทนสีเดียวกันทั้งคู่

## ไดอะแกรมประกอบ

แทรกด้วย markdown ปกติ `![คำบรรยายภาพ](/images/chNN/slug.svg)` หนึ่งบรรทัด (เว้นบรรทัดว่างล้อมรอบ)
โดยทั่วไปวางไว้หลังย่อหน้าแรก (หลักคิด/หลักการ) ก่อนเข้ารายละเอียด — ใช้ได้ทันทีทั้งเว็บ (`.md-body img`
มี CSS จัดกลาง+ขอบมนแล้ว) และ PDF (renderMd แทน `src="/images/` เป็น relative path ให้อัตโนมัติ)

### สไตล์ SVG (อ้างอิง `public/images/ch02/cathode-ray.svg`, `ch01/meniscus-reading.svg`)

- `viewBox` กว้าง ~700-760, สูงพอดีเนื้อหา (ไม่มีขอบว่างเปล่าเยอะ)
- `font-family="system-ui, sans-serif"`, ใช้ `<style>` ประกาศ class `.lbl` (label หลัก 12-13px)
  กับ `.sub` (คำอธิบายรอง 10.5-11px สีเทา `#64748b`)
- ป้ายกำกับเขียนตรงบนไดอะแกรม ไม่ใช้ legend แยกกล่อง
- โทนสี: โครงสร้าง/อุปกรณ์ = slate `#94a3b8` และฟ้าอ่อน `#f0f9ff`/`#bfdbfe`, ของเหลว/สาร = ตามจริง,
  ลูกศร/เน้นความถูกต้อง = เขียว `#059669`/`#10b981`, เตือน/ผิด = แดง `#dc2626`
- **สีแดง `#dc2626` (class `.given`) สงวนไว้สำหรับ "ค่าที่โจทย์กำหนดให้"** เท่านั้น (เช่น
  ตัวเลขที่โจทย์บอกมา) ไม่ใช้กับป้ายกำกับทั่วไป — นี่คือจุดที่หยิบยืมแนวคิด "เขียนกำกับตัวแปร
  ด้วยสีต่างจากไดอะแกรมหลัก" จากภาพตัวอย่างชีทติวฟิสิกส์ที่ผู้ใช้ส่งมา มาปรับเป็นของเราเอง
  (ห้ามคัดลอกไดอะแกรม/เลย์เอาต์ตรงจากภาพลิขสิทธิ์ของสำนักพิมพ์อื่นเด็ดขาด — ใช้เป็นแรงบันดาลใจ
  เชิงสไตล์เท่านั้น ตามหลักเดียวกับ `docs/reference-distill-process.md`)
- ตรวจเรขาคณิตก่อนใช้จริงเสมอด้วย `getBBox()` เทียบกับ `viewBox` (ดูหัวข้อถัดไป) — **ห้ามเชื่อ
  แค่สายตา** เพราะพิกัดเยื้องกันง่ายมาก โดยเฉพาะเมื่อมี `<g transform="translate(...)">` ซ้อน

### วิธีตรวจ SVG ก่อนใช้จริง (ไม่ต้องพึ่ง screenshot)

เปิดไฟล์ผ่าน Browser tool แล้วรันสคริปต์นี้ (สำคัญ: ใช้ `svg.getBBox()` ที่ตัว `<svg>` root
ตรงๆ ไม่ใช่ไล่ทีละ element ด้วยมือ เพราะ `getBBox()` ของ element ย่อยที่อยู่ใน `<g transform>`
จะคืนค่าพิกัด**ก่อน**ใส่ transform ทำให้ตรวจผิดพลาดเป็น false positive):

```js
(function(){
  const s = document.querySelector('svg');
  const vb = s.viewBox.baseVal;
  const b = s.getBBox();
  return JSON.stringify({vb:[vb.x,vb.y,vb.width,vb.height], bounds:[b.x,b.y,b.x+b.width,b.y+b.height],
    overflow: {L:b.x<vb.x, T:b.y<vb.y, R:b.x+b.width>vb.x+vb.width, B:b.y+b.height>vb.y+vb.height}});
})();
```
ถ้า `overflow` มีค่า `true` ตัวไหน แปลว่ามีเนื้อหาล้นขอบ viewBox จริง (พบแล้วจริงระหว่างทำ
บทที่ 1 — ข้อความคำบรรยายยาวเกินไปตรงกลางคอลัมน์ ทำให้ล้นซ้าย ต้องตัดให้สั้นลง/ขึ้นบรรทัดใหม่)

### แม่แบบ vs เฉพาะข้อ

โจทย์หลายข้อในหัวข้อย่อยเดียวกันมักใช้ **แนวคิดภาพเดียวกัน** ได้ (เช่น "เลือกอุปกรณ์วัดปริมาตร"
ทุกข้อใช้ `volumetric-glassware.svg` ใบเดียว) — ไม่ต้องวาดใหม่ทุกข้อ ให้ทำแม่แบบ (template) ต่อ
"แนวคิดภาพ" ไม่ใช่ต่อข้อ ส่วนโจทย์คำนวณเชิงตัวเลขที่ไม่มีภาพธรรมชาติ (เช่น เลขนัยสำคัญ, แปลงหน่วย)
ให้ใช้แม่แบบ "บันไดหน่วย/เส้นจำนวน" แล้วกำกับตัวเลขเฉพาะของข้อนั้นด้วยสีแดง (`.given`) แทนที่จะ
ฝืนวาดภาพอุปกรณ์ที่ไม่ช่วยความเข้าใจ — การตัดสินใจว่าข้อไหนควร "ใช้แม่แบบเดิม" vs "ทำสำเนา
ปรับตัวเลข" vs "วาดใหม่เฉพาะข้อ" อยู่ที่ว่าภาพนั้นต้องมี**ตัวเลขของโจทย์นั้นโดยเฉพาะ**ปรากฏใน
ภาพหรือไม่ (ถ้าใช่ → ทำสำเนาแล้วแก้ตัวเลข, ถ้าเป็นแนวคิดทั่วไป → ใช้ไฟล์เดียวกันซ้ำได้เลย)

## ผลกระทบต่อการ import Supabase (สำคัญ — อ่านก่อนรัน)

`scripts/import-questions.mjs` ทำงานเป็น**หมวดทั้งหมด (topic_id)** — ถ้าหมวดนั้นมีข้อสอบอยู่แล้ว
ในฐานข้อมูล จะ**ข้าม**เว้นแต่สั่ง `--replace` ซึ่งจะ**ลบข้อสอบเดิมทั้งหมดของหมวดนั้นก่อนลงใหม่**
(cascade ลบ `attempts` ที่ผูกกับข้อเดิมไปด้วย) — การแก้แค่ field `explanation` ของข้อที่มีอยู่แล้ว
จึงยังต้องใช้ `--replace` เสมอ ไม่มีทางเลือก "อัปเดตเฉพาะบางข้อ" ในสคริปต์ปัจจุบัน
**ต้องแจ้งผลกระทบนี้ให้เจ้าของทราบและขอยืนยันก่อนรันทุกครั้งที่แก้เนื้อหาบทที่มีข้อมูลอยู่แล้ว**
